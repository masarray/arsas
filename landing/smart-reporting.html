ARSAS handles the report engineering.

IEC 61850 reporting is powerful, but manually checking DataSets, BRCBs, URCBs, trigger options, reservations, GI, and fallback behavior can delay FAT, SAT, and commissioning. ARSAS automates the acquisition path so engineers can reach trustworthy live values without first repairing every reporting configuration issue.

Automatic acquisition path

Reporting is optimized per selected point—not treated as an all-or-nothing prerequisite.

ARSAS begins with live MMS values, proves report coverage, repairs exact gaps where allowed, and keeps fallback only where it is still required.

03

Use an association-scoped temporary DataSet and suitable available URCB/RCB where permitted.

Bounded polling fallback

Failure-aware by design

The application does not trust a report merely because an RCB exists.

Coverage must be usable or observed. ARSAS retains MMS verification and can return a point to faster polling when report traffic misses an actual change.

Configured

Usable static BRCB or URCB

Use event-driven reporting while preserving DataSet order, reasons, sequence information, quality, and IED timestamps.

Incomplete

Partial or empty coverage

Attempt dynamic recovery for the exact uncovered selected points when the IED and approved workflow permit writes.

Unavailable

No usable RCB or write permission

Continue with bounded MMS polling rather than forcing the engineer into a separate configuration tool.

Degraded

A real change is missed

Verification detects the mismatch, records diagnostics, and restores polling until event delivery is proven again.

Engineering boundary

Automatic does not mean uncontrolled.

The engineer selects the required signals. Existing static reporting is preferred. Temporary dynamic changes are association-scoped and attempted only when enabled and supported. Rejections remain visible, cleanup is attempted where permitted by the IED, and polling remains the non-blocking fallback.
