service response · completion · AddCause · feedback

A control request can be accepted before the device reports a negative final outcome.

Keep the initial service result, CommandTermination, AddCause, timing and process feedback as separate evidence. Collapsing them into one “success” or “failed” label hides the actual control path.

Immediate answer

Do not stop at the operate response.

Wait for the supported completion path, capture CommandTermination and AddCause where available, then compare the resulting status or position feedback within the approved test window.

Evidence layers
  • select response when SBO is used
  • operate service response
  • CommandTermination result
  • AddCause or detailed device reason
  • process status, position or indication feedback
Diagnostic path

Identify where the command stopped.

The client, communication service, control model, device interlocking and physical process can each produce a different outcome. Preserve the timeline.

01

Record the request contract

Keep object reference, ctlModel, command type, value, origin, checks and selected association.

02

Record immediate service result

A positive acknowledgement means the request reached a certain service stage, not necessarily that operation completed.

03

Capture final completion evidence

Retain CommandTermination, AddCause and timing instead of replacing them with a generic message.

04

Correlate process feedback

Check the resulting status, quality, timestamp and event evidence from the same IED context.

Accepted is not operated.

The service layer may accept the request while device logic later blocks or fails the operation.

Operated is not verified.

Independent status, event and process evidence remains necessary before declaring the test objective complete.
