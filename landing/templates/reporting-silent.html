MMS connects, but IEC 61850 reporting is silent.

The association can be healthy while no report arrives. Treat this as a coverage and control-block investigation—not proof that IEC 61850 communication has failed.

Follow diagnostic pathSmart Reporting
Immediate answer

Confirm the selected signal is actually covered by an enabled, usable RCB.

Check the RCB reference, linked DataSet, exact members, reservation or ownership state, enable state, trigger options and whether a general interrogation or data change is expected.

Typical symptoms
  • MMS reads work but selected values remain polling-only
  • RCB exists but its DataSet does not contain the required object
  • Report enable or reservation write is rejected
  • Another client appears to own the control block
  • Reports arrive only after GI or a real process change
Likely causes

Silent reporting usually has a specific boundary.

Common causes include empty or mismatched DataSets, occupied RCBs, unsupported writable attributes, incorrect trigger expectations, integrity periods that are longer than the observation window, or a device that exposes configured objects differently from the available SCL file.

01

Prove the value can be read

Read the target data attribute directly and retain value, quality and timestamp evidence.

02

Map exact DataSet coverage

Do not assume the Logical Node or Data Object name implies membership. Inspect the exact FCDA or live member reference.

03

Inspect RCB state

Review buffered or unbuffered type, reservation, owner, enabled state, trigger options, integrity period and configuration revision.

04

Observe the acquisition source

Distinguish report, polling and degraded fallback so the UI does not imply report success where none exists.
