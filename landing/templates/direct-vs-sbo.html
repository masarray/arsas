ctlModel · select · operate · completion

Read the live control model before choosing Direct or SBO.

The command sequence must follow the object and IED behavior actually exposed. A generic write is not a safe substitute for a typed IEC 61850 control workflow.

Direct

Operate without a separate select step

The client sends the supported operate sequence directly, subject to the live control model, origin, check conditions, timing and device logic.

  • fewer service steps
  • still requires typed command data
  • completion and feedback remain separate evidence
  • not permission to bypass project procedure
SBO

Select before operate

The client first establishes a selection context, then operates within the permitted sequence and timing window.

  • selection ownership matters
  • select and operate references must match
  • timeout or cancellation can invalidate selection
  • enhanced security may add completion evidence
Diagnostic path

Resolve the live command contract first.

Inspect ctlModel, command type, service reference, select behavior, required checks and command completion path before presenting an active control action.

01

Read ctlModel

Do not infer Direct or SBO only from the object class or vendor family.

02

Resolve typed command data

Use the live descriptor for DPC, SPC, INC, BSC, APC or other supported control form.

03

Execute the required sequence

For SBO, retain select response, timing and operate response as distinct evidence.

04

Correlate completion and feedback

A positive service response, CommandTermination and process feedback are not interchangeable.

A successful request is not proof of final operation.

Retain service result, command completion, AddCause, timing and the resulting status or position feedback.

A negative result is valuable evidence.

Do not hide rejected checks, expired selection, access denial or device interlocking behind a generic failure message.
